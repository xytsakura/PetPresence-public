# PetPresence 开源发布审计记录

更新时间：2026-07-05

这份记录用于判断 PetPresence 是否已经接近第一次公开开源发布。它只记录当前仓库状态下已经由文件、脚本和命令输出证明的事实；无法自动证明的事项仍列为人工确认项。

## 结论

当前仓库已经具备第一版公开开源的工程基础：

- 默认产品叙事已经收敛为 Agent-assisted creator pipeline，而不是实时摄像头监测系统。
- 公开 fixture 保留 synthetic `pet_demo` 和 cleaned `pet_bichon_demo`。
- Creator CLI、action scaffold、provider example、local provider import、provider result validation、透明 WebM 转换、桌面运行时和 Agent recipe 已形成闭环。
- `verify:quick`、`release:verify`、public hygiene、小白/Bichon demo smoke、公开资产审计、依赖审计、readiness gate 和 clean export smoke 均已有自动化脚本。

正式公开前仍需要人工确认：

- 仓库可见性是否从 private 切换到 public。
- 默认分支是否保留 `master` 或迁移到 `main`。
- `pet_demo` 合成素材和 prompt/config fixture 已作为 synthetic public fixture 纳入本次发布。
- `pet_bichon_demo` 的 6 个 WebM 动作已作为 cleaned public demo fixture 纳入本次发布；未来新增源视频或模型输出仍需单独审计。
- 第一版 GitHub release tag 和 release notes 是否采用 `docs/release_notes_template.md`。
- 当前大批清理和新增文件是否先按主题拆分提交，再公开仓库。
- 远端 GitHub Actions 是否在公开前通过。

## 已完成的发布守门

- `.env.example` 默认 `PETPRESENCE_PET_ID=pet_demo`。
- `smoke:bichon` 校验 `pet_bichon_demo` manifest 和桌面配置，但不打开 Electron 窗口。
- `.env.example` 不包含真实 API key。
- `.gitignore` 保持 `.env` 和 `.env.*` 私有，同时允许 `.env.example` 发布。
- `release:check-config` 防止默认宠物回退到 private legacy private legacy pet workspace。
- `release:check-hygiene` 防止 Git 可见文件或 clean export 中带出 `.env`、API key、`outputs/`、private/legacy media、frames、events 或 reports。
- `release:check-staged` 防止已经进入 Git index 的 `.env`、API key、`outputs/`、private/legacy media、frames、events、reports、key file 或 preview artifact 被提交。
- `smoke:public-hygiene` 验证 hygiene gate 会拦截危险文件，并在清理后放行。
- `release:readiness` 检查必要公开文件、package metadata、关键 npm scripts、docs index 和 GitHub Actions quick gate。
- CI 使用 `windows-latest`，安装 root 和 desktop 依赖，并运行 `npm run verify:quick`。
- `verify:quick` 可作为新贡献者和 CI 的快速验证。
- `release:verify` 可作为发布前完整本地验证。
- `release:preflight` 串联 stage plan、完整 release verification、clean export smoke、严格公开资产审计、staged 内容检查和 `git diff --check`，作为公开前最终本地总门禁。
- `release:audit-assets` 在 Git checkout 和无 `.git` 的文件系统导出中都有 fallback 行为。
- `release:smoke-clean-export` 可模拟无 `.git`、无 `node_modules`、无 private media 的 clean export。

## 当前限制

这些限制必须在 README、CHANGELOG 和 release notes 中保持清楚：

- 当前主要验证平台是 Windows。
- 当前支持开发模式 Electron 桌宠预览，不承诺安装包。
- MP4 到透明 WebM 的质量依赖源视频质量。
- 真实商业视频模型 adapter 尚未绑定到某个厂商；当前只有本地 synthetic example、local import 和 provider contract validator。
- Observer、event-server、report、QA、shelter 页面是 experimental 或 legacy，不是默认 creator pipeline。

## 人工发布前检查

在切换仓库公开或打 tag 前，人工完成：

1. 确认没有 staged/private `.env`、API key、用户素材、生成视频、frames、events、reports 或本地 `outputs/`。
2. 确认 GitHub Actions 在公开仓库通过。
3. 确认 release notes 使用当前公开 fixture 边界，不重新引入历史 workspace 清单。
4. 决定默认分支是否迁移到 `main`。
5. 决定 first tag，例如 `v0.1.0`。
6. 按 `docs/release_notes_template.md` 写 GitHub release notes。
7. 运行：

```powershell
npm run release:preflight
```

或手动运行等价命令组：

```powershell
npm run release:verify
npm run release:check-hygiene
npm run smoke:public-hygiene
npm run release:audit-assets -- --fail-on-unresolved
npm run release:smoke-clean-export
npm run release:check-staged
git diff --check
```

8. 检查 GitHub Actions 是否在远端通过。

## 建议提交粒度

当前工作树包含大量有意清理和新增文件。建议正式公开前按主题拆提交：

1. public fixture and private asset cleanup
2. creator CLI and desktop runtime alignment
3. action scaffold, Agent pipeline smoke, and quickstart smoke
4. provider adapter example, local import, and contract validation
5. release verification and asset audit tooling
6. documentation, privacy, roadmap, changelog, and release notes

不要在这些提交中重新加入 private legacy workspaces、shelter assets、presentation artifacts、frames、events、reports、原始 MP4、preview artifacts 或本地 `outputs/`。
