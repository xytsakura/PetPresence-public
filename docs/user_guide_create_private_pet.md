# 普通用户指南：用 PetPresence 做一个私人桌面宠物

这份文档给不熟悉技术细节的用户使用。你不需要先理解 Electron、WebM、ffmpeg 或视频模型，只需要准备宠物素材，并把下面的启动提示词交给你信任的 coding Agent。

PetPresence 的目标是帮你在本地做出一个私人桌面宠物。它不是默认摄像头监控系统，也不是云端宠物识别服务。

## 1. 你最终会得到什么

完成后，你会有：

- 一个可以在电脑桌面上运行的宠物小窗口。
- 几个可以切换的动作，例如 `idle`、`sleep`、`eat`、`play`。
- 一套保存在本机的宠物配置和动作素材。
- 一个 Agent 可继续维护的项目目录。

默认情况下，你的宠物照片、视频、生成素材和 API key 都不应该被提交到公开仓库。

## 2. 你需要提前准备什么

至少准备这些信息：

| 项目 | 示例 | 说明 |
| --- | --- | --- |
| 宠物名字 | 花花 | 桌宠显示名，可以是中文 |
| 宠物类型 | cat / dog / rabbit | 给 Agent 和视频模型理解用 |
| 想要的动作 | idle,sleep,eat,play | 第一版不要贪多，先做 2 到 4 个动作 |
| 宠物性格 | 安静、黏人、爱睡觉 | 用于生成气泡文案和动作提示词 |
| 素材来源 | 自己拍的视频 / 视频模型 API / 两者都有 | 决定 Agent 走哪条路径 |

推荐素材：

- 每个动作 3 到 6 秒短视频。
- 宠物主体完整，不要被遮挡。
- 背景尽量简单。
- 镜头稳定，不要快速移动。
- 不要带文字、水印和无关物体。

如果你只有照片，没有动作视频，也可以先让 Agent 生成给视频模型使用的 prompt。最终效果取决于你使用的视频生成模型是否能稳定还原你的宠物。

## 3. 复制给 Agent 的启动提示词

把下面这段话复制给你的 coding Agent，并把尖括号里的内容替换成自己的信息：

```text
请使用当前仓库 PetPresence，帮我制作一个私人桌面宠物。

我的目标不是部署实时摄像头监控，也不是云端服务；我只想做一个本地运行、可以手动切换动作的桌面宠物。

请先阅读：
- README.md
- docs/user_guide_create_private_pet.md
- docs/agent-recipes/create-your-pet.md
- docs/troubleshooting.md
- docs/media_and_data_policy.md

宠物信息：
- 名字：<宠物名字>
- 类型：<cat/dog/rabbit/bird/other>
- 性格：<几句话描述性格>
- 想要动作：<idle,sleep,eat,play 或自定义动作>
- 我已有素材：<列出本地图片/视频路径；如果没有就写“暂无动作视频”>
- 视频生成 API：<有/没有/稍后再接>
- 是否允许上传宠物素材到视频模型：<允许/不允许/每次先问我>

请按 PetPresence 的 Agent recipe 执行：
1. 检查本地环境；
2. 创建 pet_id；
3. 生成 creator_brief.md，记录我的素材来源、视频 API 状态和上传授权；
4. 让我确认 creator_brief.md；
5. 生成 action_plan.md 和 prompts；
6. 让我确认动作计划；
7. 导入已有 MP4，或为视频模型准备 prompt；
8. 注册动作素材；
9. 必要时转成透明 WebM；
10. 运行 pet:validate 和 pet:doctor；
11. 启动桌宠预览；
12. 告诉我哪些文件是私人素材，不要提交到公开仓库。

如果你要调用任何外部 API 或上传我的宠物素材，必须先停下来问我确认。
```

如果你只是想先试运行项目，可以让 Agent 先运行公开合成示例：

```text
请先不要创建我的私人宠物，先按 docs/quickstart.md 跑通 pet_demo 和 Agent pipeline smoke，并告诉我当前机器缺不缺依赖。
```

## 4. 两条素材路径

### 路径 A：你已经有动作视频

这是最稳定的路径。

你给 Agent 本地 MP4 或 WebM 文件，例如：

```text
D:\pets\huahua_idle.mp4
D:\pets\huahua_sleep.mp4
```

Agent 会把这些素材导入 PetPresence，再根据需要转换成透明背景 WebM。

适合：

- 你能拍到宠物动作视频。
- 你不想上传宠物素材到第三方模型。
- 你希望第一版尽快跑起来。

### 路径 B：你只有照片，想用视频生成 API

这是更自动化但更依赖模型质量的路径。

Agent 会先生成动作计划和 prompt，再让你选择是否把照片或参考素材上传到视频生成模型。模型生成 MP4 后，PetPresence 负责导入、透明化、注册和预览。

适合：

- 你接受使用外部视频模型。
- 你有模型 API key 或准备购买。
- 你愿意多试几次，挑质量最好的生成结果。

注意：PetPresence 开源仓库不绑定某个商业视频模型。不同模型的价格、隐私政策、还原度和使用条款都需要你自己确认。

## 5. 隐私和 API 决策

在 Agent 调用外部服务前，你应该明确回答：

| 问题 | 推荐默认值 |
| --- | --- |
| 是否允许上传宠物照片到视频模型？ | 每次先问我 |
| 是否允许上传宠物视频到视频模型？ | 每次先问我 |
| 是否允许把生成视频提交到公开仓库？ | 不允许 |
| 是否允许 Agent 保存 API key 到项目文件？ | 不允许 |
| 是否需要把最终桌宠开源成 demo？ | 先不需要 |

API key 应该放在你本机环境变量或本地 `.env` 中，不要写入 `.env.example`、README、截图、报告或公开 issue。

## 6. 第一次推荐动作组合

第一版建议只做这些：

| action | 是否必须 | 建议 |
| --- | --- | --- |
| `idle` | 必须 | 待机动作，最好能自然循环 |
| `sleep` | 推荐 | 很适合桌宠长期播放 |
| `play` | 推荐 | 用来测试动作切换 |
| `eat` | 可选 | 如果有清楚素材再做 |

不要一开始就做十几个动作。桌宠好不好看，通常取决于少数动作是否稳定、清晰、循环自然。

## 7. 你应该如何验收

Agent 说完成后，你可以按这张表检查：

| 检查项 | 通过标准 |
| --- | --- |
| 能启动桌宠 | 运行 `npm run desktop -- --pet-id <pet_id>` 后出现宠物窗口 |
| 能切换动作 | 右键菜单或命令能触发至少一个非 idle 动作 |
| 透明效果可接受 | 边缘没有严重闪烁、缺腿、缺耳朵或大片背景残留 |
| 动作表达清楚 | 一眼能看出是在睡觉、玩耍或待机 |
| 循环自然 | `idle` 和 `sleep` 不会明显跳帧 |
| 配置通过验证 | `npm run pet:validate -- --pet-id <pet_id>` 通过 |
| 隐私边界清楚 | 你知道哪些目录不能提交到公开仓库 |

如果不满意，优先让 Agent 调整或重做素材，而不是急着改复杂功能。

## 8. 常见问题

### 我完全不会写代码，可以用吗？

可以，但建议让 coding Agent 代你操作。你主要负责提供素材、确认是否允许上传到外部模型、判断视觉效果是否满意。

### 没有视频生成 API 可以用吗？

可以。你可以使用自己拍摄或剪辑的短视频。PetPresence 的核心流程不依赖付费 API。

### 能不能做猫狗以外的角色？

可以。项目并不限制物种，桌宠可以是猫、狗、兔子、鸟，也可以是玩偶、角色或物品。真正的瓶颈是视频素材或视频生成模型能不能稳定生成对应动作。

### 为什么不直接做实时摄像头监控？

实时监控需要视频流、识别模型、多模态推理和长期运行服务，成本和隐私风险都更高。PetPresence 第一版选择更轻、更可复刻的本地桌宠制作流程。

### 我可以把自己的桌宠也开源吗？

可以，但请先确认你拥有所有图片、视频、生成素材和模型输出的公开发布权。默认情况下，私人宠物素材不要提交到公开仓库。

### 跑不起来先看哪里？

先把这份故障排查指南交给 Agent：

```text
docs/troubleshooting.md
```

通常先跑 `npm run pet:doctor`、`npm run pet:validate -- --pet-id <pet_id>` 和桌面 smoke，比直接改代码更有效。

## 9. 给 Agent 的成功标准

如果你在监督 Agent，可以要求它最终说明：

- 创建的 `pet_id` 是什么。
- 已注册哪些动作。
- 哪条命令可以启动预览。
- 哪些素材来自用户，哪些来自模型生成。
- 是否调用过外部 API。
- 哪些文件不应该提交。
- `pet:doctor`、`pet:validate`、桌面 smoke 是否通过。
